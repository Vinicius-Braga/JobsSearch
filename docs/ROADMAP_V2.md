# Roadmap V2 — de script local a SaaS

A V1 (script Java que roda no seu computador, em loop, grátis) não vai mais ser usada — o esforço daqui pra frente é todo na V2: um app web com assinatura, onde a pessoa edita um perfil em texto livre, clica em "buscar vagas" sob demanda, e recebe uma lista com nota de aderência (IA) por vaga.

Decisões já tomadas:
- **Stack**: Java + Spring Boot. Reaproveita `GupyJobSource`, `JobFilter`, `Classifier` como estão — só entra uma camada web por cima.
- **Hospedagem**: em aberto, decidida na Fase 3/4. A API é desenhada de forma agnóstica de provedor até lá.
- **Modelo de IA**: Claude Haiku 4.5, pelo custo (~$0,06 por busca de ~40 vagas pré-filtradas).
- **Configuração**: todo o projeto (CLI e API) roda dentro do Spring Boot. `src/main/resources/application.yml` define a estrutura tipada (`app.telegram.*`, `app.anthropic.*`) e importa `.env` (`spring.config.import`) pra preencher os valores — nada de `env.get("CHAVE")` espalhado pelo código, cada configuração vira uma classe `@ConfigurationProperties` (`TelegramProperties`, `AnthropicProperties`) injetada onde precisa. Três entry points Spring distintos, cada um com seu próprio `@SpringBootApplication` e escopo de beans: `CliApplication` (loop de 6h, `./gradlew run`), `WebApplication` (API, `./gradlew bootRun`) e `FitScorerPlaygroundApplication` (teste manual, `./gradlew fitScorerPlayground`).

## Fase 1 — Motor como serviço

Objetivo: provar que o motor da V1 funciona sob demanda (uma chamada HTTP), não só em loop de 6h. Sem tela ainda.

- [x] Adicionar `spring-boot-starter-web` ao `build.gradle`
- [x] Criar `SearchJobsUseCase` (novo, em `application/`) — recebe lista de empresas + `JobFilter`, retorna lista de `ClassifiedJob` sincronamente. Não reutiliza `RunCycleUseCase` diretamente porque esse é do modelo de loop/notificação; o novo caso de uso é "buscar agora e devolver".
- [x] Criar `JobSearchController` em `infrastructure/web/` — `POST /api/search` recebendo `{ empresas: [...], filtro: {...} }`, devolvendo JSON com as vagas encontradas
- [x] Configurar Jackson (vem com Spring Boot) para serializar `Job`/`ClassifiedJob`
- [x] Testes de integração do endpoint com pelo menos uma empresa real (ou mock do `JobSource`)
- [x] Manter `Main.java` (modo CLI/loop) funcionando em paralelo — não quebrar o que já existe, mesmo que não seja mais usado, até termos certeza que a API cobre o caso de uso

**Critério de pronto:** dar `curl -X POST localhost:8080/api/search` com um filtro e uma empresa, e receber vagas de volta em JSON. ✅ Testado manualmente com a Vivo — retornou vaga real de RH.

> Rodar a API: `./gradlew.bat bootRun`. CLI antigo continua com `./gradlew.bat run`.

## Fase 2 — FitScorer (IA) + perfil

Objetivo: dado um perfil em texto livre e uma vaga, obter nota 1-10 + justificativa. Testado isolado, fora do fluxo real.

- [x] Criar `UserProfile` (domain) — só um texto livre por enquanto, sem persistência ainda
- [x] Criar port `application/port/FitScorer` — `score(UserProfile profile, ClassifiedJob job) -> FitScore { nota, justificativa }`
- [x] Implementar `AnthropicFitScorer` em `infrastructure/ai/` usando a API da Claude (Haiku 4.5)
- [x] Config de API key via `.env` (`ANTHROPIC_API_KEY`), no `.gitignore`, nunca commitado — mesmo arquivo passou a centralizar também as credenciais do Telegram (`TELEGRAM_TOKEN`/`TELEGRAM_CHAT_ID`, antes em `telegram.txt`)
- [x] Decidido: **chamada por vaga**, não em batch — mais simples de fazer parsing e isolar falha de uma vaga sem perder as outras; revisar pra batch só se o custo por busca (Fase 3+) se mostrar alto na prática
- [x] Teste isolado: `FitScorerPlayground` (`./gradlew.bat fitScorerPlayground`) roda perfil fixo + 4 vagas variadas contra a API real, pra validar visualmente antes de confiar no output — requer `claude.txt` configurado

**Critério de pronto:** dado um perfil e uma lista de vagas já filtradas, o `FitScorer` retorna nota + justificativa pra cada uma, sem estourar custo.

> Config: crie/edite `.env` na raiz com `ANTHROPIC_API_KEY=SUA_CHAVE_AQUI` (nunca commitado).

## Fase 3 — Tela mínima + 1 usuária de teste

Objetivo: sua namorada consegue usar o app de ponta a ponta, sem cobrança.

- [x] Thymeleaf (server-rendered) escolhido — sem abrir uma segunda stack de frontend
- [x] Login com **Spring Security** de verdade (usuário único via `InMemoryUserDetailsManager`, senha com BCrypt, credenciais no `.env`) — não é multi-usuário ainda (isso é Fase 4), mas já é o mecanismo real, não uma checagem manual
- [x] Persistência de perfil: H2 (arquivo local) via Spring Data JPA, perfil salvo por usuário logado — trocar por Postgres só quando for multi-tenant de verdade (Fase 4)
- [x] Tela: editar perfil → botão "buscar vagas" → lista de resultados com nota + link de aplicar
- [x] Ligar Fase 1 (busca) + Fase 2 (score) nesse fluxo real, ponta a ponta — `SearchAndScoreJobsUseCase`
- [x] **Removida a V1 inteira**: `CliApplication`/`CliBeanConfig`/`CliRunner`, `TelegramNotifier`/`TelegramProperties`, `NoOpNotifier`, `RunCycleUseCase`, os ports `JobRepository`/`JobPublisher`/`Notifier`, e os publishers de CSV/HTML (`CsvExporter`, `CsvJobRepository`, `CsvPublisher`, `HtmlPublisher`). O resultado agora só existe na tela, sob demanda — nada mais roda em loop nem gera arquivo

**Critério de pronto:** sua namorada edita o perfil dela, clica em buscar, e vê uma lista de vagas com nota, sem você rodar nada manualmente. ✅ **Fase 3 fechada e validada de ponta a ponta**, incluindo o caminho feliz com `ANTHROPIC_API_KEY` real: login, edição de perfil (persistiu após reiniciar o app — H2 funcionando), busca real na Gupy, pontuação real da IA (6 vagas de RH em Porto Alegre/RS, notas de 2 a 7, com justificativas coerentes sobre área/localização/porte da empresa), lista ordenada por nota, e a mensagem de aviso quando a IA falha (testado também o caminho de erro sem chave configurada).

> Limitação conhecida: a busca é síncrona — a pessoa fica esperando a página carregar enquanto a Gupy é consultada e cada vaga é pontuada uma por uma (pode levar 20-40s+ dependendo de quantas vagas passam no pré-filtro). Funciona pra 1 usuária de teste, mas vale considerar um indicador de carregamento ou processamento assíncrono antes de abrir pra mais gente (Fase 4).

## Fase 4 — Assinatura

Objetivo: abrir pra mais gente além da usuária de teste, com cobrança.

- [x] Auth multi-usuário de verdade (Spring Security + cadastro/login) — `AccountEntity`/`AccountJpaRepository`/`JpaUserDetailsService` no Postgres, cadastro em `/cadastro` (`AccountController`), senha com BCrypt. `ProfileStore` já era isolado por username desde a Fase 3, então o multi-tenant "só funcionou" ao trocar a autenticação.
- [x] Migrar de H2/SQLite pra Postgres, com isolamento de dados por usuário (multi-tenant) — banco no [Supabase](https://supabase.com/), conexão via `.env` (`DATABASE_URL`/`DATABASE_USERNAME`/`DATABASE_PASSWORD`), `ddl-auto: update` cria as tabelas (`account`, `user_profile`) automaticamente
- [x] **Checkout de pagamento (InfinitePay)**: substituído o upgrade manual via `UPDATE account SET plan='PLUS'` por um fluxo real. `GET /assinar` cria um pedido pendente (`PendingPaymentEntity`, chave `order_nsu`) e chama `POST /links` da InfinitePay (`InfinitePayGateway`), redirecionando o usuário pra página de pagamento hospedada por eles (PIX/cartão prontos — não é um QR code customizado nosso, é a página deles). Quando o pagamento é concluído, a InfinitePay chama `POST /webhooks/infinitepay` — o payload do webhook **nunca é confiado sozinho**: o handler sempre confirma com `payment_check` (`PaymentGateway.verifyPayment`) antes de promover a conta pra `PLUS` (`SubscriptionStore.upgradeToPlus`), pra evitar que uma chamada forjada ao webhook libere o plano de graça. Config nova: `INFINITEPAY_HANDLE` (identificador público da conta, não é chave secreta) e `APP_BASE_URL` (pra montar `redirect_url`/`webhook_url` — precisa ser uma URL alcançável de fora, então só funciona de verdade depois de hospedado; em `localhost` o webhook não é alcançável pela InfinitePay). `/webhooks/**` liberado no `SecurityConfig` (sem sessão/CSRF, é chamado por máquina). Testado o suficiente sem gastar dinheiro real: compilação, wiring de beans, e o webhook rejeitando `order_nsu` desconhecido sem chamar a API ao vivo — o fluxo completo (`/assinar` real + confirmação de pagamento) só pode ser validado com uma hospedagem pública ou um túnel (ex: ngrok).
- [x] **Modelo de planos definido e implementado**: `Plan` (`FREE`/`PLUS`) na `AccountEntity`, `SubscriptionStore`/`JpaSubscriptionStore`. **Grátis**: 1 busca por dia (fuso America/Sao_Paulo), só as 3 primeiras vagas aparecem — o resto some atrás de linhas "borradas" (`filter: blur()`) com um aviso de quantas vagas tem a mais, pra mostrar valor sem entregar o resultado todo. **Plus (R$5/mês, planejado)**: sem limite de busca por dia, todas as vagas aparecem sem blur. Um terceiro nível (R$15/mês) com pontuação por IA + justificativa + carta de apresentação foi definido mas fica pra depois — a UI e o modelo de dados já são compatíveis com adicionar um `Plan.PRO` no futuro. Testado de ponta a ponta: conta grátis bateu no limite de 3 vagas visíveis (57 escondidas de 60 encontradas) e foi bloqueada na segunda busca do mesmo dia; upgrade manual pra `PLUS` no Postgres liberou busca ilimitada e todas as vagas sem blur, na mesma sessão.
- [ ] **Pendência descoberta durante o teste**: `ddl-auto: update` do Hibernate tentou adicionar a coluna `plan` como `NOT NULL` sem valor padrão numa tabela que já tinha contas — Postgres rejeitou (`column "plan" of relation "account" contains null values`), e até a correção (`columnDefinition` com `default 'FREE'`), *login parou de funcionar pra qualquer conta* (erro "column does not exist" no meio do fluxo de autenticação). Isso é um risco real de `ddl-auto: update` em produção — vale considerar migrations de verdade (Flyway/Liquibase) antes de crescer mais o schema.
- [x] **Frontend mais sofisticado**: botão de busca grande com animação de radar (anéis pulsando), cinza/desabilitado sem perfil, colorido/ativo com perfil salvo, muda de cor e acelera a pulsação durante a busca. `POST /buscar` virou endpoint JSON (`BuscarResponse`) chamado via `fetch()` — a busca roda em background, sem recarregar a página, resolvendo a limitação de UX anotada na Fase 3.
- [x] **Redesign visual completo**, feito no Claude Design e implementado com fidelidade em Thymeleaf/CSS puro (sem framework JS): telas de login/cadastro/perfil em painel dividido (navy + branco), dashboard com hero em gradiente navy, card de perfil em "vidro fosco" (glassmorphism), radar reconstruído com a mecânica exata do mockup (dois anéis com borda parcial girando em direções/velocidades opostas via CSS `animation`), fonte Plus Jakarta Sans, paleta em `oklch()`. Estilos compartilhados centralizados em `fragments/styles.html`. Testado de ponta a ponta com dados reais (Postgres + IA) e visualmente idêntico ao mockup em todos os estados (login, cadastro, perfil, radar desabilitado/ativo/buscando, resultados).
- [x] **Dockerizado**: `Dockerfile` multi-stage (build com JDK, runtime com JRE só) e `docker-compose.yml`. Banco continua sendo o Postgres externo (Supabase) via `.env` — o compose não sobe banco nenhum, só a aplicação. `empresas.txt` montado como volume pra editar sem rebuild. JVM roda com `-Djava.net.preferIPv4Stack=true` — sem isso, chamadas HTTPS externas (Gupy, API da Claude) falhavam com "Network is unreachable" no Docker Desktop (contêiner sem rota IPv6 funcional). Testado: `docker compose up --build` builda e sobe o container, conecta no Postgres real, e responde `/login` corretamente.
- [x] **Removido o `filtro.txt`**: o pré-filtro (área/senioridade/região/remoto) deixou de ser um arquivo estático e virou algo que a própria IA deriva do perfil em texto livre, a cada busca. Novo port `SearchCriteriaExtractor` + `AnthropicSearchCriteriaExtractor` (Haiku) leem o perfil e devolvem um `JobFilter` (agora com campo `remoteOnly`, além de área/senioridade/região); se a extração falhar, cai pra busca sem pré-filtro (log, sem quebrar a busca). Motivo: o `filtro.txt` ficou com valores de teste antigos (RH/Porto Alegre) e um usuário real buscando "backend Java remoto" simplesmente não recebia resultado nenhum — o filtro cortava tudo antes da IA nem ver o perfil, e não existia campo de "remoto" no formato antigo. Testado de ponta a ponta com perfil real ("Desenvolvedor Backend Jr, Java, remoto, jr/pleno"): a IA extraiu área=TI e remoto=true corretamente, e a busca retornou vagas de TI remotas de verdade.
- [x] **Pontuação por IA (nota + justificativa) tirada do fluxo principal, em stand-by**: ao pensar em aumentar a lista de `empresas.txt` (hoje só ~10, muito pouco), surgiu a pergunta de como manter o custo de IA sob controle com centenas de vagas candidatas sem ter que pontuar todas. Decisão: por ora, simplificar — a busca só lista as vagas que passam no pré-filtro (extraído do perfil), sem chamar o `FitScorer` por vaga. Novo caso de uso `SearchJobsForProfileUseCase` (busca + pré-filtro, sem pontuar) é o que o `DashboardController` chama agora; `SearchAndScoreJobsUseCase`/`FitScorer`/`AnthropicFitScorer`/`ScoredJob` continuam no código, com bean registrado, prontos pra religar quando fizer sentido (ex: com uma seleção em duas etapas — 1 chamada barata de IA rankeando por metadado antes da pontuação detalhada cara, ideia que ficou anotada mas não implementada). Tabela de resultados mostra Vaga/Empresa/Local/Modalidade/Área/Senioridade em vez de Nota/Justificativa.
- [x] **Filtro por palavra-chave de stack**: bug relatado — perfil pedindo "Java" trazia vagas de .NET, porque o pré-filtro só cobria área/senioridade/região/remoto (categoria "TI" aceita qualquer stack). `AnthropicSearchCriteriaExtractor` agora também extrai `palavrasChave` (tecnologias mencionadas no perfil) e `JobFilter` ganhou o campo `keywords`, casado contra o título da vaga — sem custo extra de IA por vaga (é só um match de string, igual região já fazia). Testado: perfil "Java, remoto, jr/pleno" parou de trazer vagas .NET/QA/Data Science.
- [x] **`empresas.txt` expandido de 10 pra 50**: cada uma das 50 foi verificada com uma chamada HTTP real (`curl` contra `https://{subdominio}.gupy.io/`) antes de entrar na lista — nada de subdomínio chutado sem checar. Resultado prático: buscando "Desenvolvedor Backend Jr, Java, remoto" achou uma vaga real ("Analista de Sistemas Pleno (Java + Quarkus)" na C&A) que não existia no pool de 10 empresas antigo.
- [x] **Coleta por empresa paralelizada**: `SearchJobsUseCase` agora dispara uma virtual thread (Java 21, `Executors.newVirtualThreadPerTaskExecutor()`) por empresa em vez de esperar uma terminar pra começar a próxima — cada empresa é só espera de rede, candidato ideal pra isso. Resiliência por empresa mantida (falha de uma não derruba as outras). Testado com as 50 empresas: busca caiu de ~1min30 pra **~12s**, mesmos resultados de antes.
- [x] **Bug: "cidade ou remoto" não trazia nada**: perfil pedindo "Porto Alegre ou remoto" resultava em 0 vagas, mesmo existindo vagas remotas e vagas em Porto Alegre de verdade. Causa: `JobFilter.matches` tratava região e remoto como dois requisitos que precisam bater ao mesmo tempo (E lógico) em vez de alternativas (OU lógico) — como vaga remota normalmente não tem cidade/estado preenchidos na Gupy, exigir "remota E em Porto Alegre" nunca batia em nada. Corrigido pra região/remoto funcionarem como alternativas quando os dois são informados juntos (mantendo o comportamento estrito quando só um dos dois é pedido). Prompt do `AnthropicSearchCriteriaExtractor` também ajustado pra deixar essa semântica explícita. Testado com o perfil real relatado ("RH, estágio/júnior/assistente, Porto Alegre ou remoto"): passou de 0 pra 3 vagas reais (Renner, Sicredi, Tenda, todas em Porto Alegre).
- [x] **Segunda fonte de vagas: LinkedIn** (retirado do backlog e implementado). Novo port `LinkedInJobSource` + impl `LinkedInHttpJobSource`, no endpoint público não-autenticado `jobs-guest/jobs/api/seeMoreJobPostings/search`. `SearchJobsUseCase` mescla Gupy (por empresa) e LinkedIn (por critério de busca) no mesmo pool de virtual threads. A query usa os mesmos `keywords`/`regioes`/`remoto` que o `SearchCriteriaExtractor` já deriva do perfil — sem parâmetro novo pro usuário preencher. Parser calibrado contra HTML real do endpoint (regex sobre `data-entity-urn`, `base-search-card__title/__subtitle`, `job-search-card__location`), com teste automatizado fixando essa estrutura. Retry com backoff exponencial em 429/5xx (até 4 tentativas). **Bug real encontrado testando**: `location=Brasil` (português) faz o LinkedIn *ignorar* o filtro de local e devolver vagas dos EUA — só `location=Brazil` (inglês) funciona pro termo genérico de país; cidade/estado específicos em português funcionam normalmente. Testado de ponta a ponta com perfil real ("Backend Java, remoto, jr/pleno"): foi de 1 vaga (só Gupy) pra 6 vagas reais (1 Gupy + 5 LinkedIn — PagBank, TOTVS, Instituto Recôncavo, Unimed, Enghouse Systems). **Risco mantido do backlog original**: scraping não-oficial, contra os Termos de Serviço do LinkedIn, mais frágil que a Gupy — volume por busca já é limitado (no máx. 2 páginas por query, no máx. 2 queries por busca).
- [x] **UI do modelo de planos**: dashboard ganhou selo de plano (FREE/PLUS) ao lado da logo, link "Ver planos" que abre um modal comparando Free (R$0) vs Plus (R$5/mês, preço vindo do `DashboardController`, não hardcoded no template) com botão "Assinar PLUS" (mostra aviso de "em breve" — Stripe ainda não integrado). Estado de limite diário atingido (`limitReached`, calculado no controller) agora desabilita o radar mostrando "Limite atingido" (tanto no carregamento da página quanto ao vivo após um clique que bate no limite) e troca o aviso de texto solto por uma caixa estilizada com CTA pro modal de planos. O antigo bloco de linhas borradas por vaga trancada (`tr.locked-row`, `filter: blur()`) foi substituído por um único card de upsell abaixo da tabela ("Encontramos mais N vaga(s) — visíveis no plano PLUS"). Mudança puramente visual — a pontuação por IA (nota/justificativa) segue desligada do fluxo, como já estava. Testado de ponta a ponta com dados reais: busca retornando 6 vagas (3 visíveis + upsell de 3), segunda busca no mesmo dia mostrando o estado de limite atingido, e modal de planos abrindo tanto pelo "Ver planos" quanto pelos CTAs do limite/upsell.
- [x] **Hospedagem decidida: Render** (plano free — sem cartão de crédito, 750h/mês, builda direto do `Dockerfile`). `render.yaml` (Blueprint) criado na raiz do repo pra deploy reprodutível: secrets (`DATABASE_URL`/`DATABASE_USERNAME`/`DATABASE_PASSWORD`/`ANTHROPIC_API_KEY`/`INFINITEPAY_HANDLE`) marcados `sync: false` (preenchidos manualmente no dashboard, nunca commitados). `APP_BASE_URL` não precisa ser configurada no Render — `application.yml` agora cai pra `RENDER_EXTERNAL_URL` (injetada automaticamente pelo Render) quando `APP_BASE_URL` não está setada, só usando o literal `http://localhost:8080` como último fallback pra dev local. `Dockerfile` ganhou `-XX:MaxRAMPercentage=75.0` — o padrão da JVM (25% da RAM do container) é conservador demais pro plano free (512MB, único processo no container). Risco identificado, não testado ainda: o Render monitora e pode suspender serviços free com "volume incomum de tráfego de saída" — o app chama ~50 empresas Gupy + LinkedIn + Anthropic em paralelo (virtual threads) a cada busca, o que pode acionar esse radar; vale observar depois do primeiro deploy real. Deploy em si (conectar o repo, preencher os secrets no dashboard) ainda não foi feito — depende de ação manual do usuário no Render.

**Testado de ponta a ponta:** cadastro de conta nova → login → perfil isolado por usuário (conta nova sem dados da conta anterior) → perfil salvo persiste no Postgres real (Supabase) → busca assíncrona real (Gupy + IA) retorna 6 vagas ordenadas por nota via JSON, renderizadas pelo JS sem reload — confirmado via `curl` com sessão autenticada e visualmente no navegador (radar ativo → estado de busca → resultados), incluindo o redesign visual completo.

## Ordem de trabalho

As fases são sequenciais por dependência técnica (2 depende de 1 pra ter vagas pra pontuar; 3 depende de 1+2 pra ter algo pra mostrar; 4 depende de 3 funcionando de ponta a ponta com uma usuária real). Não pular fase, mas dentro de cada fase os itens de checklist podem ser feitos fora de ordem quando fizer sentido.
